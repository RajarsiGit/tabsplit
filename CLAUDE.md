# CLAUDE.md

This file provides guidance to Claude Code when working with code in this repository.

## Project Overview

TabSplit is a lightweight alternative to Splitwise for splitting recurring shared expenses (roommates, trips). Vite + React frontend, Tailwind CSS, Vercel serverless functions, Neon Postgres. Architecture mirrors `smart-task-manager` (cookie-based JWT auth, `@neondatabase/serverless`, one handler file per resource under `api/`). Licensed under GPL-3.0-or-later (see `LICENSE`).

## Commands

```bash
npm run dev          # Vite dev server on http://localhost:5173
npm run dev:api      # vercel dev - serves api/* functions (needs `vercel link` + env pulled)
npm run build        # Production build to dist/
npm run preview      # Preview production build
```

**`npm run dev` proxies `/api/*` to the live deployed API** (see `server.proxy` in `vite.config.js`), not to a local `vercel dev` instance. The browser only ever talks to `localhost:5173`, so it's same-origin and the auth cookie works without any cross-origin cookie/CORS handling — but it also means local frontend work hits the **production** database by default. Point that proxy target back at `http://localhost:3000` and run `npm run dev:api` alongside it if you want a fully local stack instead.

```bash
DATABASE_URL=postgresql://...   # Neon Postgres connection string
JWT_SECRET=your_secret_key      # Secret for JWT signing
CRON_SECRET=                    # Set automatically by Vercel Cron; used to authorize api/cron/process-recurring
GITHUB_CLIENT_ID=               # GitHub OAuth App, for "Continue with GitHub" login

BLOB_READ_WRITE_TOKEN=          # Vercel Blob store token, for expense receipt uploads
```

### Neon role/ownership gotcha

This project's `DATABASE_URL` connects as a non-owner Postgres role (e.g. `tabsplit`), while the tables live under the database's default `neondb_owner` role. A fresh `CREATE TABLE` run through the app's connection is owned by the creating role and works fine (including `CREATE INDEX` on it), but **`ALTER TABLE` on any of the original tables** (`users`, `groups`, `expenses`, etc.) **fails with `must be owner of table X`** because the app role only has DML grants (SELECT/INSERT/UPDATE/DELETE) on those, not ownership. Schema changes that alter an existing table (as opposed to creating a new one) need to be run from the Neon Console's SQL Editor (which runs as the owner role), not via `psql $DATABASE_URL`. `schema/permissions.sql` grants the app role DML/USAGE/EXECUTE rights on everything the owner role creates (existing objects plus a default-privileges rule for future ones) — run it once, also from the Neon Console, after applying `schema.sql` on a fresh database.

Concretely, `groups.archived_at` (added for group archiving) needed the Neon Console; `group_categories` and `expense_comments` (both brand-new tables) applied fine via plain `psql $DATABASE_URL -f schema/schema.sql`. Same story for the budgets/itemized-splits/multi-currency batch: `budgets`, `expense_items`, and `expense_item_participants` (brand-new tables) applied fine via plain `psql` (or an equivalent script using `@neondatabase/serverless`'s `Client`, when `psql` itself isn't installed locally); `recurring_expenses.end_date` and `expenses.original_amount`/`original_currency` needed the Neon Console. When adding a column to an existing table, add the matching standalone `ALTER TABLE ... ADD COLUMN IF NOT EXISTS ...` statement and call out in your response that it needs to run from the Neon Console (the `CREATE TABLE IF NOT EXISTS` block alone is a no-op against an already-existing table, so it won't retroactively add the column). Once that Console run has actually landed against the live database, it's fine — and preferred for readability — to fold the column directly into `schema.sql`'s `CREATE TABLE IF NOT EXISTS` block afterward and drop the standalone `ALTER TABLE` line: a fresh database created from `schema.sql` gets the column either way (inline, in this case), and the already-migrated live database already has it, so nothing is lost by inlining. Don't skip the Console step itself and jump straight to inlining, though — inlining only updates what a *future* fresh install gets, not the database you're actually running against.

## Data Model

- `users` — id, name, email, password (bcrypt hash, nullable for GitHub-only accounts), github_id (nullable, unique)
- `groups` — a household or trip; `currency` defaults to INR; `archived_at` (nullable) hides a group from the default groups list without deleting it — set/cleared via `POST /api/groups?id=X&action=archive`
- `group_members` — join table, `role` is `owner` or `member`
- `expenses` — one-off charges; `recurring_expense_id` set when materialized from a template; `paid_by` is nullable (NULL for multi-payer expenses — see `expense_payments`); `receipt_url` optional (Vercel Blob URL); `split_type` is `equal`, `exact`, `percentage`, or `itemized` (see `expense_items`); `original_amount`/`original_currency` are nullable and **purely informational** — a display-only tag for "this was actually paid in a different currency" (e.g. "$50 USD"). `amount` (in the group's own currency) is always the value splits/payments/balances key off; nothing in the app converts `original_amount` into `amount` or vice versa
- `expense_splits` — per-user share of an expense; rows are replaced (not updated) whenever an expense's split changes
- `expense_payments` — per-payer contribution to an expense (supports splitting the *payment* itself across more than one payer, e.g. two roommates who split paying for one grocery run); every expense has at least one row here, even single-payer ones — this table, not `expenses.paid_by`, is the source of truth for "who paid what" (see Balance Calculation)
- `expense_comments` — free-text comments on an expense (`expense_id`, `user_id`, `body`, `created_at`); any group member can post one, only the author can delete their own
- `expense_items` / `expense_item_participants` — the line-item breakdown behind a `split_type: "itemized"` expense (e.g. splitting one grocery receipt item by item, where different people were in on different items). `expense_items` is `expense_id`, `description`, `amount`; `expense_item_participants` is a join table (`item_id`, `user_id`) mirroring `expense_splits`/`expense_payments`'s normalized style rather than a JSON array column. These tables are **not** a second source of truth for balances — `api/expenses.js`'s `aggregateItemizedSplits(items)` sums each item's own equal split across its participants into one total per user, and that aggregate is what actually gets written to `expense_splits` (via the normal `writeSplits` path) and is what `computeBalances` reads
- `recurring_expenses` — templates (rent, subscriptions); always split **equally across all current group members** at generation time — there's no per-template participant list; stay single-payer (materialized `expenses` rows still get exactly one `expense_payments` row); `active` toggles whether the cron job still generates from it (pause/resume, doesn't delete the template); `end_date` (nullable) auto-deactivates the template once its next occurrence would fall after it — see `api/cron/process-recurring.js`
- `settlements` — manual "I paid you back $X" records between two members
- `group_invites` — one active invite token per group (`UNIQUE` on `group_id`); regenerating replaces the old token outright
- `notifications` — in-app notifications (`user_id`, `group_id`, `type`, `message`, `read_at`); also doubles as the data source for each group's **Activity** feed (`GET /api/notifications?groupId=X&scope=group`, any member) — see the note under `api/notifications.js` below for the tradeoffs of that reuse
- `group_categories` — custom expense categories a group has added on top of the built-in defaults in `src/utils/categories.js` (`CATEGORIES`); `expenses.category`/`recurring_expenses.category` stay free-text VARCHAR, so this table only feeds dropdown options for that group — it's not a foreign key and nothing enforces an expense's category exists here
- `budgets` — a monthly spending limit, whole-group (`category IS NULL`) or per-category, per group. `UNIQUE (group_id, category)` plus a partial unique index (`WHERE category IS NULL`) enforce at most one whole-group budget and at most one budget per category per group — see the "Common Pitfalls" note on Postgres treating `NULL` as distinct under a plain `UNIQUE` constraint. `last_notified_at` debounces the `budget_exceeded` notification to once per calendar month per budget; `period` exists for future non-monthly support but only `monthly` is implemented

Schema lives in `schema/schema.sql`. Apply it directly against the Neon database (`psql $DATABASE_URL -f schema/schema.sql`) — there's no migration runner.

## Backend Architecture (`api/*.js`)

One file per resource, method-based routing inside a single `export default handler`, following the reference project's pattern.

**Vercel's Hobby plan caps a deployment at 12 Serverless Functions, and every file directly under `api/` counts as one — even pure helper modules with no `export default` handler.** Shared helpers that aren't themselves an endpoint (`db.js`, `balances.js`, `recurrence.js`, `format.js`) live in `api/_lib/` instead: Vercel ignores any file or directory under `api/` whose name starts with `_` when counting/deploying functions, so they're still importable via relative paths but don't burn a function slot. Keep new shared helpers in `api/_lib/`; only files meant to be routable endpoints belong directly in `api/`. **Currently at 11/12** — prefer folding a new sub-resource into an existing handler via an `?action=` branch (as `groups.js` does for `members`/`archive`/`categories` and `expenses.js` does for `comments`) over adding a new top-level file, unless it's a genuinely new resource.

- `api/_lib/db.js` — `getDb()` (lazy Neon client), `requireAuth(req)` (verifies JWT cookie, throws if missing/invalid), `requireGroupMember(sql, groupId, userId)` (throws if not a member), cookie helpers (auth cookie + a short-lived OAuth `state` cookie for CSRF), CORS helper
- `api/_lib/format.js` — `formatCurrency(amount, currency)`, a backend-side mirror of the frontend's `src/utils/categories.js` helper of the same name (kept separate since `api/` and `src/` are different bundles); used by the settle-up reminder cron for notification message text
- `api/auth.js` — `/api/auth/register|login|logout|me`, routed via `?path=` query param through `vercel.json` rewrites (see reference project's auth.js for the same pattern). Also `/api/auth/github` (redirects to GitHub's authorize URL) and `/api/auth/github/callback` (exchanges the code, finds-or-creates/links a user by `github_id`/email, sets the auth cookie, redirects to `/`). The callback never returns JSON errors — any failure redirects to `/?authError=...` since it's a full-page browser flow, not a fetch call. `me` also returns `has_password`/`has_github` booleans (never the password hash) so the UI knows which account-settings actions are safe to offer. `PUT /api/auth/profile` updates `name`; `PUT /api/auth/password` changes the password (or sets one for a GitHub-only account, skipping the current-password check when none is set yet); `POST /api/auth/github/unlink` disconnects GitHub but is blocked if the account has no password (would lock the user out). The `github/callback` handler also doubles as the **link** flow: if a valid session cookie is already present when it runs (i.e. the user started from the Settings page while logged in, not the login screen), it attaches the GitHub identity to that session's user instead of doing find-or-create/login, and redirects to `/settings?linked=github` (or `/settings?authError=...` on failure) instead of `/`.
- `api/groups.js` — group CRUD, membership (add by email lookup, remove), returns computed `balances` + `settleUp` suggestions on `GET ?id=`; notifies a member when added or when their role changes. Also handles, via `?action=`: `archive` (`POST ?id=X&action=archive`, body `{ archived }`, owner-only — sets/clears `archived_at`; `GET /api/groups` excludes archived groups unless `?includeArchived=true`), `categories` (`GET`/`POST ?id=X&action=categories` for any member to list/add a custom category, `DELETE` owner-only — see `group_categories`), and `budgets` (`GET ?id=X&action=budgets` for any member to list; `POST`/`PUT`/`DELETE` owner-only to create/update/remove — see `budgets`). The `POST`/`PUT` budget handlers do a manual `SELECT`-then-`UPDATE`-or-`INSERT` upsert rather than `ON CONFLICT`, since the whole-group case (`category IS NULL`) is backed by a partial unique index that a plain `ON CONFLICT (group_id, category)` clause can't target.
- `api/expenses.js` — expense CRUD; owns the split-calculation logic (`splitEqually`, `splitByPercentage`, exact-split validation) and the payer-calculation logic (`writePayments`, which validates a `payments: [{userId, amount}]` array sums to the total the same way `writeSplits` validates exact splits); a single `paidBy` in the request body is normalized into a one-entry `payments` array (`normalizePayments`) so every expense — single- or multi-payer — always has `expense_payments` rows; accepts optional `receiptUrl`, `originalAmount`/`originalCurrency` (passed straight through, purely additive — see `expenses` in the Data Model); notifies other group members when a new expense is added; exports `splitEqually` for reuse by the cron job. `splitByPercentage` uses largest-remainder apportionment (same rounding-remainder-distribution idea as `splitEqually`, weighted by each participant's `percentage` instead of splitting evenly) for `splitType: "percentage"`. `splitType: "itemized"` works differently from the other three: the client sends an `items: [{description, amount, participantIds}]` array instead of (or in addition to) `participants`, and the server — not the client — computes `participants` via `aggregateItemizedSplits(items)` before calling the same `writeSplits` every other split type goes through; `writeItems` then separately delete-then-inserts `expense_items`/`expense_item_participants`. Also handles, via `?action=`: `comments` (`GET`/`POST ?id=X&action=comments` for any group member to list/add a comment — posting notifies the other members — `DELETE ?id=X&action=comments&commentId=Y` restricted to the comment's own author), `items` (`GET ?id=X&action=items`, any member — the itemized breakdown for one expense), and `export` (`GET ?groupId=X&action=export`, any member — a `text/csv` download of that group's expenses, one row per expense). A local (non-exported) `checkBudgetsAndNotify(sql, groupId, category)` runs after every `POST`/`PUT` write (wrapped in try/catch so a budget-check failure never blocks the expense write itself) — see `budgets` in the Data Model and pitfall 12 below.
- `api/recurring.js` — recurring expense template CRUD (stays single-payer, no `payments` array); `PUT` already supported toggling `active` (pause/resume) before the UI exposed it — no API change was needed to add that to `GroupDetail`'s Recurring tab and the cross-group `AllRecurring` page. `POST`/`PUT` also accept an optional `endDate` → `recurring_expenses.end_date`.
- `api/settlements.js` — record/undo manual settlements (undo has no restriction beyond group membership — any group member can undo any settlement, not just its recorder or participants); notifies the counterparty (whichever of `fromUser`/`toUser` isn't the actor) when a settlement is recorded. Also handles `POST ?action=nudge` (body `{groupId, from, to, amount}`) — a one-off reminder notification (`type: "settle_up_nudge"`) sent to `from`, for nudging someone about a specific `settleUp` suggestion outside the weekly cron reminder; any group member can nudge any pairing, matching that same looseness.
- `api/_lib/balances.js` — `computeBalances(sql, groupId)` (net balance per user, "paid" side summed from `expense_payments` — not `expenses.paid_by` — so multi-payer expenses attribute correctly; plus splits + settlements) and `simplifyDebts(balances)` (greedy min-transaction settle-up suggestions). Both operate purely on `expenses.amount`, which is always single (group-)currency by construction (see `original_amount`/`original_currency` above) — there is no cross-currency debt-simplification problem to solve here.
- `api/account.js` — always operates on the caller's own account. `GET /api/account` returns a full JSON export (profile, groups, expenses, expense splits, recurring templates, settlements) with a `Content-Disposition: attachment` header, for the Settings page's "Export my data" link (also surfaced directly in the sidebar's user menu). `DELETE /api/account`, body `{ mode: "associated" | "own" }`: `associated` hard-deletes the user row (groups they solely own are deleted outright first; everything else cascades per `schema/schema.sql`'s `ON DELETE CASCADE` rules). `own` hands off sole-owned groups to the longest-tenured other member (or deletes the group if they're the only member), removes the user from every `group_members` row, then scrubs the `users` row in place (name/email/password/github_id) instead of deleting it, so shared expenses/splits/settlements they were part of stay visible to other members.
- `api/_lib/recurrence.js` — `nextOccurrence(dateStr, frequency)` date math (weekly/monthly, monthly clamps to end-of-month)
- `api/cron/process-recurring.js` — Vercel Cron target (see `vercel.json` `crons`, runs daily at 06:00 UTC). Materializes any `recurring_expenses` due (`next_occurrence <= CURRENT_DATE`, and `end_date IS NULL OR next_occurrence <= end_date`) into real `expenses` rows split across current group members (writing a matching single-payer `expense_payments` row too), then advances `next_occurrence` and auto-sets `active = FALSE` once the *next* occurrence after this one would fall past `end_date`. Protected by `CRON_SECRET` if set.
- `api/cron/settle-up-reminders.js` — second Vercel Cron target, runs weekly (Mondays 09:00 UTC). For every group, computes balances and runs `simplifyDebts`, then sends each debtor a `settle_up_reminder` notification naming who they owe and how much. Same `CRON_SECRET` gate as `process-recurring.js`. (Distinct from the manual, one-off `settle_up_nudge` in `api/settlements.js` — this one is automatic and covers every debtor in every group, not a specific `{from, to}` pairing a user picked.)
- `api/invites.js` — one active invite token per group. `POST ?groupId=X` (owner-only) generates/regenerates the token (upsert on `group_invites`, which has a `UNIQUE` constraint on `group_id`); `DELETE ?groupId=X` (owner-only) revokes it; `GET ?groupId=X` (owner-only) fetches the current token for display; `GET ?token=X` (any authenticated user, deliberately **not** gated by `requireGroupMember` — that's the point) previews the group's name/member count; `POST ?token=X&action=accept` joins the group (idempotent if already a member).
- `api/notifications.js` — exports `createNotification(sql, { userId, groupId, type, message, read = false })` for reuse by other handlers (the `read` flag inserts with `read_at` already set, for **self-logged audit entries** — see below), plus `GET /api/notifications?limit=X` (defaults to 30, capped at 200 — the bell dropdown uses the default, the full-page `Activity` view asks for 200), `POST ?action=read&id=X`, `POST ?action=read-all`. Also `GET ?groupId=X&scope=group` (any member) — the shared, group-wide **Activity** feed (`GroupActivity.jsx`, in a `GroupDetail` group's Settings tab), which reuses this same table rather than a dedicated audit-log table. The endpoint collapses the near-duplicate rows a multi-recipient event produces (one `INSERT` per recipient, same `type`/`message`) via `SELECT DISTINCT ON (type, message, created_at)`.

  **Every mutation across the app logs the acting user's own action**, not just the handful of types that notify *other* people about it (that used to be a real gap — an actor's own actions didn't show up in "their" own group's feed — closed by adding one more `createNotification` call, with `read: true`, at every mutation site that lacked one; see `api/groups.js`, `api/expenses.js`, `api/recurring.js`, `api/settlements.js`, `api/invites.js`, `api/auth.js`, `api/account.js`). `read: true` keeps these self-logged entries out of the unread bell badge (`read_at IS NULL` is what drives that count) while still showing up in both the personal Activity page and, since `?scope=group` filters by `group_id` alone, the shared group feed. Two phrasing/placement rules to preserve when adding a new call site: group-scoped self-logs must be **third-person** (`"${actorName} did X"`, never `"You..."` — other members read the same shared feed and a false "You" would be confusing); account-level actions (profile, password, GitHub link/unlink, data export, account-deletion self-entries) use `groupId: null` and first-person, since they only ever render on the acting user's own personal Activity page. See pitfall 15 for the one real gap this reuse still has (deleting the thing a log entry would reference).
- `api/blob-upload.js` — implements `@vercel/blob/client`'s `handleUpload` for expense receipt uploads; requires auth, restricts to image content-types, 8MB cap. The frontend uploads directly to Blob via this handler (never proxies the file bytes through a Vercel Function), then passes the resulting URL as `receiptUrl` to `api/expenses.js`.

**Auth**: JWT in an HttpOnly cookie (`auth_token`, 7-day expiry — longer than the reference project's 1h since this isn't a task manager people re-open constantly). `requireAuth` throws; every handler catches and returns 401. The cookie is `SameSite=None; Secure` (not `Lax`) — required together, since browsers reject `SameSite=None` without `Secure` — chosen to support cross-origin dev setups. `Secure` cookies still work over plain `http://localhost`/`127.0.0.1`: browsers treat those as "potentially trustworthy" origins per the Secure Contexts spec, so this doesn't break local dev. **Security tradeoff to be aware of**: `setCors` (`api/_lib/db.js`) reflects whatever `Origin` header the caller sends and sets `Access-Control-Allow-Credentials: true`, with no origin allowlist. Combined with `SameSite=None`, that means literally any website can make credentialed requests to this API and read the JSON responses (a CORS-based CSRF exposure) — tightening `setCors` to an explicit allowlist (production domain(s) + localhost) is a known follow-up, not yet done.

**Authorization**: nearly every group-scoped endpoint calls `requireGroupMember` before touching data — a user must be a member of a group to read or write anything in it. Owner-only actions (editing group name/description/currency, deleting a group, changing another member's role) go through `requireGroupOwner`. `isSoleOwner(sql, groupId, userId)` (in `api/_lib/db.js`) guards against demoting or removing a group's last remaining owner — promote another member first.

## Balance Calculation

Net balance per user = (amount they paid across all `expense_payments`) − (their share across all `expense_splits`) + (settlements they sent) − (settlements they received). Positive = owed money; negative = owes money. `simplifyDebts` reduces the full pairwise picture to a minimal set of `{from, to, amount}` transactions using a greedy largest-debtor-to-largest-creditor match — this is what powers the "Settle up" suggestions in the UI, not a literal transaction-by-transaction ledger.

## Frontend Architecture (`src/`)

- `context/AppContext.jsx` — auth state only (`user`, `login`, `register`, `logout`), hydrated from `/api/auth/me` on load
- `context/ThemeContext.jsx` — dark mode. `theme` is `"light" | "dark" | "system"`, persisted to `localStorage` (`tabsplit-theme`); a `useEffect` toggles a `dark` class on `document.documentElement` (Tailwind's `darkMode: "class"` in `tailwind.config.js`) and, when `theme === "system"`, subscribes to `matchMedia("(prefers-color-scheme: dark)")` changes. `useTheme()` exposes `{ theme, setTheme }`. Wraps the app above `AppProvider` in `main.jsx`.
- `utils/api.js` — thin fetch wrapper per resource (`authApi`, `groupsApi`, `expensesApi`, `recurringApi`, `settlementsApi`, `invitesApi`, `notificationsApi`, `accountApi`), all requests use `credentials: 'include'`
- `utils/categories.js` — `CATEGORIES` (the built-in defaults), `formatCurrency`, and `mergeCategories(customCategories)` (dedupes a group's custom categories against the defaults) — see `group_categories` in the Data Model
- `App.jsx` — route-aware auth gating, not a single top-level gate: every route decides for itself whether it needs a logged-in user. `/invite/:token` renders `AuthScreen` when logged out and `AuthedShell + InviteAccept` when logged in — same URL either way, so logging in through an invite link continues straight to joining the group with no stashed-redirect logic. `/*` renders `AuthedShell + MainRoutes` (`/` → `GroupsList`, `/groups/:id` → `GroupDetail`, `/expenses` → `AllExpenses`, `/settle-up` → `SettleUp`, `/recurring` → `AllRecurring`, `/activity` → `Activity`, `/settings` → `AccountSettings`) when logged in, `AuthScreen` otherwise. `AuthedShell` is the `<Sidebar/> + <main>` wrapper shared by both branches; `<main>` carries `sm:ml-64` to offset the sidebar, which is `sm:fixed` (pinned full-height, not a flex sibling) so it doesn't scroll away with page content.
- `components/Sidebar.jsx` — replaces the old top navbar. Desktop (`sm:` and up): a `fixed` left column (`SidebarContent` — brand/bell header, nav links, `GroupSwitcher`, `UserMenu`). Below `sm`: a normal-flow top `<header>` (hamburger + brand + bell) plus a hamburger-triggered slide-over drawer (also `SidebarContent`, but with `showBrandHeader={false}` — it renders a plain "close" button instead of its own brand+bell row, since the mobile header behind it already has one; duplicating both there produced two overlapping bells/logos through the dimmed backdrop). `UserMenu` holds the avatar/name/email trigger, the light/dark/system `Appearance` toggle, and links to Settings, data export, and logout.
- `components/GroupSwitcher.jsx` — collapsible "Your groups" list in the sidebar for jumping directly to a group without going through the dashboard; fetches `groupsApi.list()` (so archived groups are excluded by default, same as the dashboard).
- `components/NotificationsBell.jsx` — badge + dropdown, polls `/api/notifications` every 30s; clicking a notification with a `group_id` marks it read and navigates to that group. Takes an `align` prop (`"left" | "right"`, default `"right"`) for which edge the dropdown anchors to — it's wider than the sidebar column, so the sidebar/drawer instance passes `align="left"` (opens rightward into the main content area) while the standalone mobile-header instance keeps the default (opens leftward, since it sits near the right edge of a full-width bar).
- `components/GroupsList.jsx` — the `/` dashboard: `Dashboard` stats, then the "Your groups" list + create-group form, then a collapsible archived-groups section (`showArchived` toggle, fetches `groupsApi.list(true)` on demand and filters to `archived_at`-set rows) with per-owner Archive/Unarchive actions.
- `components/AllExpenses.jsx`, `SettleUp.jsx`, `AllRecurring.jsx`, `Activity.jsx` — cross-group pages reachable from the sidebar, each fetching `groupsApi.list()` then fanning out a per-group request (`expensesApi.listForGroup`, `groupsApi.get` for `settleUp`, `recurringApi.listForGroup`, or `notificationsApi.list(200)` respectively) and flattening the results into one view, since none of those resources have a native cross-group endpoint. `AllExpenses` derives its category filter options from whatever categories actually appear in the loaded expenses (covers custom categories without a separate fetch), and also has `fromDate`/`toDate` inputs filtering the same `filtered` `useMemo`; `SettleUp` and `AllRecurring` group results back by group name since debts/recurring templates aren't fungible across groups/currencies. (`Activity` here is the personal notification history behind the bell — distinct from the per-group `GroupActivity` feed below.)
- `components/GroupDetail.jsx` — tabbed view (Expenses / Recurring / Balances / Insights / Members / Settings) for a single group; owns data loading for all tabs. The Expenses tab has search/category-filter/sort/date-range controls (`visibleExpenses`, derived via `useMemo`), an "Export CSV" link (`expensesApi.exportUrl(id)`), and per-row actions: an inline expand-to-view comments section (`ExpenseComments`, one open at a time via `expandedCommentsId`), an inline expand-to-view items section for `split_type: "itemized"` rows (`ExpenseItems`, via a separate `expandedItemsId` so both can't collide), a Duplicate button (fetches the full expense via `expensesApi.get`, stores it in `duplicatingExpense`, and renders `<AddExpenseForm duplicateFrom={...}>` — deliberately not `expense={...}`, since that prop is what switches the form into edit/update mode), and Edit/Delete as before. ## Common Pitfalls

1. **Recurring expenses have no participant list** — they always split equally across whoever is a group member when the cron job runs, not whoever was a member when the template was created. Adding/removing members changes future recurring splits.
2. **Split amounts must reconcile to the penny.** `splitEqually` (in `api/expenses.js`) distributes rounding remainder cents to the first N participants; exact splits are validated to sum to the total within 1 cent.
3. **Group membership check runs before every mutation.** If you add a new group-scoped endpoint, call `requireGroupMember` first or the authorization story breaks.
4. **Removing a member is blocked while they have a non-zero balance** (see `api/groups.js` DELETE `?action=members`) — settle up first.
5. **`expense_splits` rows are replaced, not patched**, on any split-affecting update — always delete-then-insert, never try to diff old vs. new splits.
6. **Neon tagged-template queries only** — build queries with `` sql`...${value}...` ``, never interpolate raw strings into the query text.
7. **`expense_payments` rows are replaced, not patched**, on any payer-affecting update — same delete-then-insert rule as `expense_splits`. Every expense must have at least one `expense_payments` row (even single-payer ones) or `computeBalances` will silently undercount what that expense's payer(s) are owed.
8. **A group's invite token is singular** (`group_invites.group_id` is `UNIQUE`) — regenerating overwrites the existing row via `ON CONFLICT (group_id) DO UPDATE`, it doesn't create a second active link.
9. **Route dispatch inside a handler is first-match, not most-specific-match.** Each `api/*.js` file is a chain of `if (req.method === X && ...)` blocks checked top to bottom. A generic block like `if (req.method === "GET" && id)` will swallow a more specific `if (req.method === "GET" && id && action === "categories")` request if the generic block appears first — the specific block never runs. Always put `action=`-guarded branches for a method *before* that method's unguarded/generic handler (this bit us for real: `groups.js`'s category and archive endpoints were added after their generic counterparts and silently never executed, crashing the frontend with a type error instead of returning the expected shape).
10. **Don't use `defaultProps` on function components** — React logs a deprecation warning (removal planned for a future major version). Use a default parameter in the destructured props instead: `function Foo({ bar = "default" })`, not `Foo.defaultProps = { bar: "default" }`.
11. **Cross-group pages (`AllExpenses`, `SettleUp`, `AllRecurring`, `Activity`) have no dedicated backend aggregation endpoint** — they call `groupsApi.list()` then fan out one request per group client-side and flatten the results. Fine at the group counts this app expects; if that ever becomes a real bottleneck, the fix is a proper backend aggregation endpoint, not more client-side fan-out.
12. **Itemized splits don't bypass `writeSplits`.** `splitType: "itemized"` still goes through the same `writeSplits(sql, expenseId, splitType, amount, participants)` every other split type uses — the server just computes `participants` first, via `aggregateItemizedSplits(items)`, instead of taking it from the client. `expense_items`/`expense_item_participants` are a separate, sibling write (`writeItems`) and are never read by `computeBalances` — if you touch `writeSplits`'s shared validation/insert logic, re-test `equal`/`exact`/`percentage` too, since they all share it.
13. **`expenses.original_amount`/`original_currency` are display-only.** Nothing converts between them and `amount` — not on write, not in `computeBalances`, not in `simplifyDebts`. If a future change makes them do real currency conversion, that's a materially different feature (needs an FX rate source), not a small tweak to the existing fields.
14. **A plain `UNIQUE` constraint doesn't stop duplicate `NULL`s.** `budgets` has `UNIQUE (group_id, category)`, but Postgres treats every `NULL` as distinct from every other `NULL`, so that alone would let a group end up with multiple whole-group (`category IS NULL`) budgets. A partial unique index (`... WHERE category IS NULL`) closes that gap — and because a partial index isn't a full-table constraint, `ON CONFLICT (group_id, category)` can't target it either, hence the manual `SELECT`-then-`UPDATE`-or-`INSERT` upsert in `api/groups.js`'s budget `POST` handler instead of a one-line `ON CONFLICT DO UPDATE`.
15. **A log entry can't outlive the row it's attached to.** `notifications.group_id` and `notifications.user_id` are both `ON DELETE CASCADE`. Logging "this group was deleted" (or "this account was hard-deleted") tied to that same `group_id`/`user_id` gets cascade-removed the instant the delete happens — it never actually persists, even though the `INSERT` itself succeeds. The pattern used everywhere this comes up (`api/groups.js`'s `DELETE` group handler, `api/account.js`'s `deleteAssociatedRecords`): insert those specific entries with `groupId: null` instead, naming the group in the message text, sent to whoever *survives* the delete (other members losing access, not the group or user row being removed). There is no workaround for logging a hard-deleted user's own "you deleted your account" action to themselves — their `user_id` is gone too, so even a `groupId: null` self-entry cascades away. That's accepted as a schema-level limitation, not a bug to chase.
